## 2. 제품 목적 및 배경

### 해결하려는 문제

프리랜서와 소규모 에이전시는 견적서를 이메일이나 PDF 첨부 파일로 전달하는데,
이 방식은 수정 시마다 재발송이 필요하고 클라이언트 입장에서는 최신본을 식별하기 어렵다.
노션을 이미 업무 도구로 사용하는 팀이 별도 견적서 소프트웨어 없이
노션 데이터베이스만으로 전문적인 견적서를 웹으로 공유할 수 있는 방법이 필요하다.

### 목표 사용자 (Persona)

**Admin** (프리랜서 / 에이전시 담당자)
노션을 일상적으로 사용하며, 별도 SaaS 도입 없이 기존 워크플로우 안에서 견적서를 발행·관리하고 싶은 1~10인 규모 팀.

**Client** (클라이언트 / 발주사 담당자)
링크 하나로 견적 내용을 확인하고 PDF로 보관하길 원하며, 별도 앱 설치나 회원가입 없이 바로 열람 가능한 환경을 선호하는 비기술 직군.

### 비즈니스 목표

- 노션 기반 워크플로우를 그대로 유지하면서 클라이언트 대면 품질을 높인다.
- 견적 발행에서 클라이언트 확인까지의 마찰을 최소화한다.
- MVP를 통해 핵심 사용자 가치를 검증하고 이후 전자서명·결제 연동 등 V2 기능의 필요성을 판단한다.

## 3. 범위 정의

### MVP In-Scope

| # | 기능 |
|---|------|
| 1 | 노션 데이터베이스 연동 및 견적 데이터 조회 |

| 7 | Admin 대시보드 — 견적서 목록 조회 |

### MVP Out-of-Scope

| 기능 | 제외 이유 | 비고 |
|------|----------|------|

| 견적서 직접 편집 UI | 노션이 편집 도구, 중복 개발 회피 | Out |

## 4. 기능 요구사항

#### F-001. 노션 데이터베이스 연동 및 견적 데이터 조회

- **우선순위**: P0 (필수)
- **설명**: Notion API를 통해 지정된 데이터베이스에서 견적서 데이터를 서버 사이드에서 조회한다.
- **사용자 스토리**: "나는 Admin으로서, 노션에 입력한 견적 데이터가 웹에서 실시간으로 반영되길 원한다. 왜냐하면 별도 데이터 입력 없이 기존 워크플로우를 유지하고 싶기 때문이다."
- **인수 조건(AC)**:
  - [ ] AC-1: `NOTION_API_KEY`와 `NOTION_DATABASE_ID` 환경변수가 설정되면 데이터베이스 조회가 성공한다.
  - [ ] AC-2: 노션 데이터베이스의 각 페이지가 견적서 1건으로 매핑된다.
  - [ ] AC-3: API 오류 시 사용자에게 친화적인 오류 메시지가 표시된다.
  - [ ] AC-4: 서버 컴포넌트에서 `fetch` + `cache: 'no-store'` 또는 적절한 revalidate 설정으로 최신 데이터를 반영한다.
- **기술 구현 힌트**: `src/lib/notion.ts`에 `@notionhq/client` 래퍼 함수 작성. Next.js Route Handler 또는 서버 컴포넌트에서 직접 호출. API Key는 서버 사이드에서만 노출.

---

#### F-002. 견적서 고유 URL 생성 (UUID 슬러그 기반)

- **우선순위**: P0 (필수)
- **설명**: 각 견적서에 예측 불가능한 고유 식별자를 부여하여 공개 URL을 생성한다.
- **사용자 스토리**: "나는 Admin으로서, 클라이언트에게 공유할 고유 링크를 얻고 싶다. 왜냐하면 특정 클라이언트만 해당 견적서를 볼 수 있어야 하기 때문이다."
- **인수 조건(AC)**:
  - [ ] AC-1: 노션 페이지 ID를 기반으로 `/invoice/[pageId]` 형태의 URL이 생성된다.
  - [ ] AC-2: 존재하지 않는 ID로 접근 시 Next.js `notFound()`를 반환한다.
  - [ ] AC-3: URL에 견적서 제목 등 민감 정보가 포함되지 않는다.
- **기술 구현 힌트**: 노션 페이지 UUID를 슬러그로 직접 활용 (`app/invoice/[pageId]/page.tsx`). 별도 DB 없이 노션 ID만으로 라우팅 가능.

---

#### F-003. 견적서 웹 뷰어 페이지 (공개 접근)

- **우선순위**: P0 (필수)
- **설명**: 인증 없이 URL만으로 접근 가능한 견적서 전용 뷰어 페이지. 브랜드 로고, 발행사 정보, 클라이언트 정보, 견적 내용을 포함한다.
- **사용자 스토리**: "나는 Client로서, 링크 클릭만으로 견적서를 바로 확인하고 싶다. 왜냐하면 회원가입이나 앱 설치 없이 바로 내용을 검토해야 하기 때문이다."
- **인수 조건(AC)**:
  - [ ] AC-1: 로그인 없이 `/invoice/[pageId]` URL 접근 시 견적서가 정상 렌더링된다.
  - [ ] AC-2: 모바일(375px 이상)과 데스크탑(1280px 이하) 해상도에서 레이아웃이 깨지지 않는다.
  - [ ] AC-3: 페이지 로딩 시 LCP가 3초 이내이다 (Vercel Edge 기준).
  - [ ] AC-4: 메타 태그(og:title, og:description)가 견적서 제목 기반으로 생성된다.
- **기술 구현 힌트**: `app/invoice/[pageId]/page.tsx`에서 `generateMetadata` 함수로 OG 태그 생성. 뷰어 레이아웃은 `(dashboard)` route group과 별도로 분리.

#### F-004. 견적서 항목 표시

- **우선순위**: P0 (필수)
- **설명**: 견적서의 핵심 정보인 품목 테이블(품목명, 수량, 단가, 금액)과 소계·세금·최종금액을 표시한다.
- **사용자 스토리**: "나는 Client로서, 각 항목의 금액과 합계를 명확하게 확인하고 싶다. 왜냐하면 견적 내용을 검토하여 승인 여부를 결정해야 하기 때문이다."
- **인수 조건(AC)**:
  - [ ] AC-1: 품목 테이블에 품목명, 수량, 단가, 금액(수량×단가) 컬럼이 표시된다.
  - [ ] AC-2: 소계, 세금(부가세 10% 기준), 최종금액이 테이블 하단에 표시된다.
  - [ ] AC-3: 금액은 한국 원화 형식(₩1,000,000)으로 포맷된다.
  - [ ] AC-4: 품목이 없는 견적서의 경우 "항목 없음" 빈 상태 메시지가 표시된다.
  - [ ] AC-5: 품목이 10개 이상일 경우에도 스크롤 없이 전체가 인쇄 영역에 포함된다.
- **기술 구현 힌트**: 노션 데이터베이스의 하위 항목(relation 또는 별도 DB)을 조회하거나, 품목을 멀티셀렉트/텍스트 파싱으로 처리. `Intl.NumberFormat('ko-KR', { style: 'currency', currency: 'KRW' })` 사용.

#### F-005. PDF 다운로드

- **우선순위**: P0 (필수)
- **설명**: 클라이언트가 견적서를 PDF 파일로 저장할 수 있는 기능. MVP에서는 `window.print()` + 프린트 최적화 CSS로 구현한다.
- **사용자 스토리**: "나는 Client로서, 견적서를 PDF 파일로 다운로드하여 사내 결재에 첨부하고 싶다. 왜냐하면 디지털 문서로 보관해야 하기 때문이다."
- **인수 조건(AC)**:
  - [ ] AC-1: [PDF 다운로드] 버튼 클릭 시 브라우저 인쇄 대화상자가 열린다.
  - [ ] AC-2: `@media print` CSS로 네비게이션, 버튼 등 불필요한 UI 요소가 인쇄 시 숨겨진다.
  - [ ] AC-3: 인쇄 결과물이 A4 세로 기준 1~2장 내에 맞게 레이아웃된다.
  - [ ] AC-4: Chrome, Safari, Edge 최신 버전에서 정상 동작한다.
- **기술 구현 힌트**: `"use client"` 버튼 컴포넌트에서 `window.print()` 호출. `globals.css`에 `@media print { .no-print { display: none; } }` 추가. 향후 V2에서 `react-pdf` 또는 Puppeteer 기반 서버사이드 PDF 생성으로 교체 가능하도록 버튼 인터페이스 추상화.

#### F-006. 견적서 상태 표시

- **우선순위**: P1 (높음)
- **설명**: 견적서의 현재 처리 상태를 색상 배지로 시각화한다.
- **사용자 스토리**: "나는 Client로서, 이 견적서가 어떤 상태인지 한눈에 알고 싶다. 왜냐하면 승인됐는지, 아직 검토 중인지 파악해야 하기 때문이다."
- **인수 조건(AC)**:
  - [ ] AC-1: 상태는 `초안(Draft)`, `발송됨(Sent)`, `승인됨(Approved)`, `거절됨(Rejected)` 4가지이다.
  - [ ] AC-2: 각 상태별로 구분되는 색상 배지(shadcn/ui `Badge`)가 표시된다.
  - [ ] AC-3: 노션 셀렉트 속성의 값이 변경되면 다음 페이지 방문 시 즉시 반영된다.
- **기술 구현 힌트**: 노션 `Select` 속성으로 상태 관리. 상태-색상 매핑 객체를 `src/lib/invoice.ts`에 정의. `Badge` 컴포넌트의 `variant` prop 활용.

- **인수 조건(AC)**:
  - [ ] AC-1: `/dashboard/invoices`에서 노션 DB의 모든 견적서 목록이 표시된다.

  - [ ] AC-4: 각 행 클릭 시 해당 견적서의 공개 뷰어 페이지로 이동한다.

- **인수 조건(AC)**:
  - [ ] AC-1: 견적서 목록의 각 행에 [링크 복사] 버튼이 표시된다.
  - [ ] AC-2: 버튼 클릭 시 `navigator.clipboard.writeText()`로 공개 URL이 복사된다.

## 5. 비기능 요구사항

### 성능

| 지표 | 목표 |
|------|------|
| 견적서 뷰어 LCP | ≤ 3초 (Vercel Edge, 한국 리전) |
| Admin 대시보드 초기 로드 | ≤ 2초 |
| Notion API 응답 캐싱 | `revalidate: 60` (1분 ISR) |

### 보안

- 공개 견적서 URL은 노션 페이지 UUID(128bit)를 그대로 사용 → 추측 불가
- `NOTION_API_KEY`는 서버 사이드 환경변수로만 관리, 클라이언트 번들에 포함 금지
- Admin 대시보드는 MVP에서 별도 인증 없음 → 내부 배포(Vercel private) 또는 환경변수 기반 Basic Auth 미들웨어 적용 권장
- Rate limiting: Vercel의 기본 DDoS 보호 활용

### 반응형

| 브레이크포인트 | 지원 여부 |
|---------------|----------|
| 모바일 (375px+) | ✅ 지원 |
| 태블릿 (768px+) | ✅ 지원 |
| 데스크탑 (1280px+) | ✅ 지원 |

### 접근성

- 색상 대비 WCAG AA 기준 준수 (4.5:1 이상)
- 모든 버튼에 `aria-label` 속성 부여
- 키보드 네비게이션 가능 (Tab 순서 보장)
- 이미지 사용 시 `alt` 속성 필수

## 6. 데이터 모델 (Notion 데이터베이스 스키마)

### 메인 DB: 견적서 (Invoices)

| 속성명 | Notion 타입 | 설명 | 필수 여부 |
|--------|------------|------|----------|
| `제목` | Title | 견적서 제목 (예: "2026년 4월 웹사이트 개발 견적") | ✅ |
| `클라이언트명` | Text | 수신 클라이언트 회사명 또는 담당자명 | ✅ |
| `상태` | Select | `초안` / `발송됨` / `승인됨` / `거절됨` | ✅ |
| `발행일` | Date | 견적서 발행 날짜 | ✅ |
| `유효기간` | Date | 견적서 유효 만료 날짜 | ⬜ |
| `공급사명` | Text | 발행자(Admin) 회사명 | ✅ |
| `공급사 연락처` | Email | 발행자 이메일 | ⬜ |
| `메모` | Text | 클라이언트에게 전달할 추가 안내사항 | ⬜ |
| `소계` | Number | 품목 합계 금액 (세전) | ✅ |
| `세금` | Number | 부가세 금액 (소계 × 10%) | ✅ |
| `최종금액` | Number | 소계 + 세금 | ✅ |

### 서브 DB: 견적 항목 (Invoice Items)

| 속성명 | Notion 타입 | 설명 | 필수 여부 |
|--------|------------|------|----------|
| `품목명` | Title | 제공 서비스/상품명 | ✅ |
| `견적서` | Relation | 메인 DB(견적서)와 연결 | ✅ |
| `수량` | Number | 수량 | ✅ |
| `단가` | Number | 단위 가격 (원화) | ✅ |
| `금액` | Formula | `수량 × 단가` (자동 계산) | ✅ |
| `설명` | Text | 품목 상세 설명 | ⬜ |

> **MVP 단순화 옵션**: 서브 DB 구성이 복잡한 경우, 품목 데이터를 메인 DB의 `Rich Text` 속성에 JSON 또는 구분자(pipe) 형태로 저장하고 파싱하는 방식으로 단순화 가능.

## 9. 오픈 이슈 및 의사결정 필요 항목

| # | 이슈 | 옵션 A | 옵션 B | 권장 |
|---|------|--------|--------|------|
| I-01 | PDF 생성 방식 | `window.print()` + CSS (간단, 무료) | `react-pdf` 또는 서버사이드 Puppeteer (고품질, 복잡) | **MVP: 옵션 A**, V2에서 옵션 B 도입 |
| I-02 | 견적 항목 저장 방식 | 별도 서브 DB (Relation 연결) | 메인 DB Rich Text 파싱 | **MVP: 옵션 B** (단순화), V2에서 옵션 A로 전환 |
| I-03 | Admin 접근 제어 | 인증 없음 (내부 배포) | Next.js 미들웨어 Basic Auth | **MVP: 옵션 A** → 내부 팀만 접근, 추후 옵션 B |
| I-04 | 데이터 캐싱 전략 | `cache: 'no-store'` (항상 최신) | ISR `revalidate: 60` (1분 캐시) | **권장: 옵션 B** (성능 + 신선도 균형) |
| I-05 | 견적서 URL 식별자 | 노션 페이지 UUID 직접 사용 | 별도 짧은 코드 생성 (DB 필요) | **MVP: 옵션 A** (DB 없이 단순화) |
| I-06 | 모바일 PDF | `window.print()` 모바일 지원 제한 | 스크린샷 안내 메시지 표시 | **MVP: 옵션 B** (모바일에서 안내 배너) |
